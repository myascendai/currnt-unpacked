# Data model

The Currnt MySQL database holds **314 tables and approximately 140 GB on disk**. About 95% of the volume lives in eight table clusters; the other ~300 tables average a few hundred MB each. This page tours the eight clusters that matter and points out the table-level oddities worth knowing.

The engine is MySQL 5.5 (years past EOL ‚Äî see [/security](/currnt-unpacked/security)). The character set is `utf8mb3` for legacy reasons; full-UTF-8 (`utf8mb4`) is on the someday list but applies only to a handful of newer tables.

## Cluster 1 ¬∑ `prospects` and friends

The CRM&apos;s main fact table. About **15 million rows** and ~20 GB.

- `prospects` ‚Äî one row per identified contact. Columns include identity (name, email, LinkedIn URL when known), enrichment (employer, title, seniority guess, location), engagement (last contacted, last opened, last replied), and a long tail of source/import metadata.
- `prospect_companies` ‚Äî denormalized employer table; updated by enrichment runs.
- `prospect_segments` ‚Äî many-to-many mapping of prospects to internal segments.
- `prospect_dedup_candidates` ‚Äî rows generated by the dedup-run handler; staff resolves through Treehouse.
- `campaign_membership` ‚Äî many-to-many prospects ‚Üî outreach campaigns.

<Callout type="note" title="The deduplication rule">
  The dedup algorithm is a multi-pass match by: (1) exact email; (2)
  case-folded email-domain plus fuzzy first/last name; (3) LinkedIn URL when
  present. Candidates surface to a human, never auto-merge.
</Callout>

## Cluster 2 ¬∑ `users`, `roles`, sessions

The unified user identity model.

- `users` ‚Äî every account: visitor, member, expert, client, staff. Role distinction is by flag and `member_type`.
- `user_alt_emails` ‚Äî secondary emails accumulated by the account-merge flow.
- `roles`, `user_roles` ‚Äî for the very small set of staff permission tiers.
- Sessions live in Redis, **not** in MySQL ‚Äî but a `session_audit` table records significant lifecycle events (login, suspend, force-logout).

<Callout type="warning" title="users.password sized for SHA-256">
  The password column is a CHAR(40), sized for a hex-encoded SHA-256 digest.
  Migrating to bcrypt or argon2 was on the roadmap; never landed. Covered on
  the [security postmortem](/currnt-unpacked/security).
</Callout>

## Cluster 3 ¬∑ `boards`, posts, transcripts

The product data model.

- `boards` ‚Äî one row per engagement (board or sprint).
- `board_members` ‚Äî many-to-many users ‚Üî boards, with role on the board (expert, member, observer, staff).
- `posts` ‚Äî every authored message in any board. Rich-text body stored as HTML.
- `transcripts` ‚Äî full conversation exports, regenerated on demand.
- `summaries` ‚Äî analyst-written deliverables, one or more per board.

## Cluster 4 ¬∑ LLM trace table

A single table accumulating **about 12 GB of LLM call/response logs** from the AI co-facilitator process. Each row records prompt, response, model, latency, and the board/post that triggered the call.

This table is unusually large for its row count because individual rows can be very long (a full prompt plus a multi-paragraph response). It is also write-only ‚Äî nothing reads from it at runtime ‚Äî and is the cleanest deletion target in the database if disk reclamation ever becomes urgent.

## Cluster 5 ¬∑ `dispenserd_jobs` and worker traces

The queue&apos;s persistence layer.

- `dispenserd_jobs` ‚Äî one row per enqueued job: type, lane, payload, status, attempts, error, timestamps.
- `dispenserd_worker_heartbeats` ‚Äî periodic check-ins from each worker process.
- `dispenserd_dead_jobs` ‚Äî overflow from `dispenserd_jobs` once a job is marked dead (this is a view, not a separate table).

## Cluster 6 ¬∑ `campaigns`, `email_sends`, tracking

Outreach&apos;s history.

- `campaigns` ‚Äî one per outreach blast; carries template id, segment id, scheduling.
- `email_sends` ‚Äî one row per per-recipient send; the largest table after prospects, ~80 M rows.
- `email_events` ‚Äî opens, clicks, bounces, complaints.
- `smtp_accounts` ‚Äî the rotation pool, with per-account health stats.

## Cluster 7 ¬∑ `invoices`, `payouts`, ledger

Operations data.

- `invoices` ‚Äî one row per client invoice.
- `invoice_line_items` ‚Äî line breakdowns.
- `payouts` ‚Äî expert compensation rows.
- `ledger_entries` ‚Äî append-only ledger of every money-affecting event.
- (Dormant) `gnosis_safe_txns` ‚Äî historical multi-sig transfers; nothing recent.

## Cluster 8 ¬∑ `blog_posts`, `assets`, content

The content-production data store.

- `blog_posts` ‚Äî about a thousand rows; published, draft, scheduled.
- `blog_post_revisions` ‚Äî versioned history.
- `assets` ‚Äî uploaded files (images, PDFs).
- `marketing_copy`, `marketing_copy_revisions` ‚Äî landing-page text and email-campaign templates.

## The long tail

The remaining ~280 tables fall into three buckets:

1. **Lookup tables** ‚Äî countries, industries, seniority levels, board templates.
2. **Audit and event logs** ‚Äî most under 1 GB each; collectively ~10 GB.
3. **Dormant feature tables** ‚Äî schema still intact from features that no longer run (Calendly bookings, Lusha enrichments, NDA acceptances, MetaMask signatures). Several have zero rows since 2022 or 2023.

A condensed inventory of the dormant subset is on the [sunset page](/currnt-unpacked/sunset).

## Schema management

No migration tool was ever adopted (no Liquibase, no Flyway, no Alembic-equivalent). Schema changes were applied manually as `ALTER TABLE` statements run by staff during low-traffic windows. The codebase has a `migrations/` directory but it stopped getting updated around 2019.

The implication: there is **no canonical schema-as-code**. The database is the schema. Reading the codebase&apos;s model files (`models/*.js` in the Sails app) gets you close but not exact ‚Äî fields have drifted, and several "legacy" columns that the models don&apos;t reference are still present in the actual tables.
