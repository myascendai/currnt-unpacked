# Architecture

Currnt was a **SailsJS monolith** — a single Node.js process tree running on a single VM, talking to a second VM that hosted the database. After nine years of growth it was still recognizably the same shape. No microservices, no Kubernetes, no message broker beyond a homegrown HTTP-poll queue. The simplicity is part of the story.

## The application layer — `http02`

A single VM (`http02`, 10.240.0.25 / 130.211.163.74) runs the entire user-facing application. Nginx fronts five PM2-managed Node workers on port 9999. There is no second app VM. There is no autoscaler. Deploys are a manual `git pull` on `master` followed by a `pm2 reload` — no CI, no PR, no feature branches.

```
                          ┌───────────────────────────┐
        traffic ─────────▶│ nginx (5 workers, 80/443) │
                          └───────────────┬───────────┘
                                          │
                                          ▼
                          ┌───────────────────────────┐
                          │ pm2 cluster (5 workers)   │
                          │ SailsJS on :9999          │
                          └───────────────┬───────────┘
                                          │
                  ┌───────────────────────┼───────────────────────┐
                  ▼                       ▼                       ▼
              MySQL 5.5             Elasticsearch             Redis
              (db01:3306)           2.4 (db01:9200)           (sessions,
                                                              socket.io)
```

The workflow visible in `git log` is **direct commits to `master`** with the message `progress` — no PR discipline, no feature branches. The frozen `HEAD` is `192a21e` (Dec 18, 2024, "Update open_ai.js").

## The data layer — `db01`

A second VM (`db01`, 10.240.0.6) hosts both **MySQL 5.5** and **Elasticsearch 2.4**. Both are years past end-of-life. The host OS only offers `ssh-rsa` / `ssh-dss` host key types — modern OpenSSH rejects it without explicit downgrade flags. The schema is 314 tables and ~140 GB of data, dominated by:

- **`prospects`** — 15 million rows. The CRM/outreach engine&apos;s main table.
- **`gptlog`** — 12 GB. Logs from the AI co-facilitator (see `/features/ai-cofacilitator`).
- **`logs_*`** — assorted activity logs.
- **`users`** — ~hundreds of thousands of accounts; password column is sized for SHA-256 (40 chars), not bcrypt.

Redis runs on `db01:6379` and is used for Sails session storage and Socket.io pub/sub between the cluster workers.

## The queue — `dispenserd`

Async work doesn&apos;t go through Kafka, RabbitMQ, SQS, or Redis pub/sub. It goes through `dispenserd`, a **custom HTTP-poll queue** that lives in the codebase. Producers `POST` jobs to it; long-running `worker0X` VMs poll it for work over HTTP and execute handlers. There are 5 lanes (priority classes), 13 worker processes managed by Supervisord, and 19 distinct handler scripts.

```
              ┌──────────────────────────────────────────┐
              │            dispenserd (HTTP)             │
              │   5 lanes · in-memory queue · MySQL ack  │
              └────────────┬─────────────────────────────┘
                           │ (poll)
       ┌───────────────────┼───────────────────┐
       ▼                   ▼                   ▼
   worker01            worker02            worker03 …
   (Supervisord)       (Supervisord)       (Supervisord)
       │                   │                   │
       └──── 19 handlers ──┴───── 4,591-line ──┘
                                 worker entry
```

A future page (`/architecture/dispenserd`) will go deeper on lane semantics, retry policy, and the handler-by-handler topology. For MVP this lives here as a placeholder.

## The cron layer

Crontab on `http02` runs **62 distinct scheduled jobs**, ranging from the obvious (digest email batches, metric rollups) to the very specific (`gpt_ai_cofac_nudging`, which every five minutes wakes the AI co-facilitator process to draft promotional posts under `ai_user_id=1`). The full list lives in the PRD; a curated cut will appear at `/architecture/cron`.

## The email layer

177 email templates live in `views/emails/`. Sending goes through a rotating fleet of SMTP accounts plus Mandrill as a fallback. The Mandrill API key was found in plaintext in `/root/check_up` on `castle-new`; it&apos;s referenced categorically on the [security page](#) (link forthcoming in phase 3) without quoting the value.

## What&apos;s on `castle-new`

A third VM, `castle-new` (10.240.0.20), exists in the project but is **half-decommissioned**. No nginx, no Node, no app. Only `sshd` listens. Bash history shows it was used as a jumpbox to reach `http02` / `db01` / the worker fleet. Old artifacts (`cert_2016/`, `cert_2017/`, `cert_wc_2018/`, `dc_to_gcloud.log` from the 2021 datacenter migration) suggest it was historically the SSL certificate and ops box. It can probably be shut down without production impact, after a forgotten-secrets audit.
